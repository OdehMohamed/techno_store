## Active task

**In progress — "Current Application Review & Evolution," reviewing Auth & Entry.** This is the phase agreed to follow Product Documentation: going workflow-by-workflow through the existing app, sequenced by product area (Auth & Entry → Reception & Maintenance → Admin → Customer/Retail later), asking of each one whether the business logic is correct, whether it aligns with `docs/product/PRD.md`, whether it has bugs/edge cases/security concerns, architectural or technical debt, dead code, and whether it's ready for implementation or needs more discovery/design/architecture first. A Foundation Design Decisions pass (dark/light mode, Material 3, Arabic-first, WCAG 2.2 AA, staff/customer visual registers, motion) is complete.

The first full workflow review — Splash → Sign-In → Phone Verification → Profile Completion → Reset Password → Forced Update — surfaced two decisions that needed resolving before the review could continue (staff account creation was found to have no live implementation at all; `ADR-004` and `PRD.md` were found to directly contradict each other about the `isActivated` field). Both are now settled and reconciled into `docs/product/` via PR #9 (`d3ada95`; see `DECISIONS_LOG.md`, 2026-07-23 entry).

**Still outstanding from that same review pass, not yet resolved:** a likely navigation bug stranding new phone sign-ups on the OTP screen after successful verification; a phone country-code propagation bug in the sign-in form; a silent password-reset failure; dead email/password-auth-adjacent code (now partially relevant again given the staff-account decision); and two seemingly unreconciled app-update mechanisms (the documented `AppUpdateCubit`/`ForcedUpdatePage` system vs. an undocumented Android-only `in_app_update` flow in `SignIn`). The review resumes from these findings next.
