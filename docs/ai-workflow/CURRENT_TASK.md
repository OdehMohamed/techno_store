## Active task

**In progress — "Current Application Review & Evolution," reviewing Auth & Entry.** This is the phase agreed to follow Product Documentation: going workflow-by-workflow through the existing app, sequenced by product area (Auth & Entry → Reception & Maintenance → Admin → Customer/Retail later), asking of each one whether the business logic is correct, whether it aligns with `docs/product/PRD.md`, whether it has bugs/edge cases/security concerns, architectural or technical debt, dead code, and whether it's ready for implementation or needs more discovery/design/architecture first.

**Staff Auth workflow behavior is now fully settled** (PR #11, `584bd27`): entry point, dedicated email/password screen, error handling, password reset within that flow, status enforcement at sign-in/restart/session, and the session/device model (multi-device allowed, no default sign-out timeout, immediate forced sign-out on deactivation or role change). Shared-device identity switching/locking is registered as its own open item — a confirmed need with no designed mechanism yet. See `DECISIONS_LOG.md` (2026-07-23 entries) for this and the two earlier reconciliations this same review pass produced (PR #9 `d3ada95`, PR #10 `75b735c`).

**Next: the Staff Status Architecture Pass**, before anything is treated as implementation-ready. Scope per product owner: the staff-status data model, write authority, Firestore security rules, live-session enforcement mechanism, behavior on role/status change at the code level, migration from the legacy `isActivated` field, and failure handling if status can't be verified. Not yet started as of this entry.
